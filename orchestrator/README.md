# orchestrator

The Rust state machine that is the robot's brain. It spawns and coordinates the Python ML
workers via stdin/stdout JSON IPC, manages the voice listening lifecycle, drives audio
playback, and will control GPIO/motors on the Raspberry Pi.

## States

```
Booting → Ready → Activating → Listening → Classifying
                                                ├── get butter       → Scanning → Approaching → Delivering
                                                ├── other intents    → Responding
                                                └── existential crisis / unknown → ExistentialCrisis

Responding      ──► Ready
ExistentialCrisis ──► Ready
Delivering      ──► Ready  (or Scanning if butter dropped)
```

The voice worker is only active during `Listening` state. All other states have the
microphone off, which prevents the robot from hearing its own audio clips.

## Dependencies

Install Rust via [rustup](https://rustup.rs):

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
```

The Rust dependencies (`serde`, `serde_json`, `rand`) are fetched automatically by Cargo.

Audio playback uses `afplay` on macOS and `mpg123` on Linux — both are standard and
require no extra installation on Raspberry Pi OS.

## Build

```bash
cargo build --release
```

## Run

```bash
cargo run
```

Or the release binary directly:

```bash
./target/release/orchestrator
```

The orchestrator must be run from the `orchestrator/` directory so that relative paths to
`../butterbot_voice/` and the Python workers resolve correctly.

Both Python workers are spawned automatically on startup. Once both report ready, the
terminal will print:

```
[State] Ready — press Enter to activate
```

Press **Enter** to simulate the physical activation button. On the Raspberry Pi this will
be replaced by a GPIO button press.

## On the Raspberry Pi

When GPIO support is added, the button listener in `main.rs` will be replaced with an
`rppal` GPIO interrupt. Add `rppal` to `Cargo.toml`:

```toml
rppal = "0.14"
```

The `systemd` service file should set `WorkingDirectory` to the `orchestrator/` directory
so relative paths continue to resolve correctly.
